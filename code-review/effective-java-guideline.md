# Effective Java 기반 리뷰 가이드

이 문서는 Joshua Bloch의 Effective Java(3판)가 제시하는 자바 작성 원칙을 코드 리뷰 점검 항목으로 정리한 추가 가이드다.
모든 자바 변경에 적용한다.

각 항목은 책의 원칙을 바탕으로 하되, 설명과 예시는 이 가이드에서 새로 작성했다.
원칙을 책에서 직접 확인할 수 있도록 관련 항목 번호를 함께 적어 두었으니, 자세한 근거와 배경은 원서를 참고한다.
이 문서는 책의 본문이나 코드를 옮긴 것이 아니라, 책의 원칙을 점검 기준으로 재구성한 것이다.

## 1. 객체 생성과 파괴

점검 항목
* 생성자 대신 정적 팩터리 메서드라는 선택지를 검토했는가 (아이템 1)
  이름으로 의도를 드러낼 수 있고, 호출마다 새 인스턴스를 만들지 않아도 된다.
* 생성자 인자가 많으면 빌더를 고려했는가 (아이템 2)
  인자 순서 혼동을 줄이고 어떤 값을 넣는지 이름으로 드러낸다.
* 자원을 직접 명시하지 않고 의존성 주입으로 받는가 (아이템 5)
  Spring에서는 생성자 주입으로 협력 객체를 받아 결합을 낮추고 테스트를 쉽게 한다.
* 불필요한 객체 생성을 피했는가 (아이템 6)
  반복마다 같은 불변 객체를 새로 만들지 않고 재사용한다.
* 다 쓴 참조를 정리해 메모리 누수를 막는가 (아이템 7)
  컬렉션이나 캐시가 참조를 계속 들고 있으면 GC 대상이 되지 못한다.
* try-finally 대신 try-with-resources로 자원을 닫는가 (아이템 9)
  예외가 나도 자원을 확실히 해제하고 코드도 간결해진다.

```java
// 생성자 주입으로 의존성을 받음 (아이템 5)
@Service
public class OrderService {
    private final PaymentClient paymentClient;

    public OrderService(PaymentClient paymentClient) {
        this.paymentClient = paymentClient;
    }
}
```

## 2. 모든 객체의 공통 메서드

점검 항목
* equals를 재정의할 때 규약(반사성, 대칭성, 추이성, 일관성)을 지켰는가 (아이템 10)
* equals를 재정의하면 hashCode도 함께 재정의했는가 (아이템 11)
  둘이 어긋나면 HashMap이나 HashSet에서 객체를 못 찾는 문제가 생긴다.
* 디버깅에 도움이 되도록 toString을 의미 있게 재정의했는가 (아이템 12)

```java
// equals와 hashCode를 함께 재정의 (아이템 10, 11)
@Override
public boolean equals(Object o) {
    if (this == o) return true;
    if (!(o instanceof Member member)) return false;
    return Objects.equals(id, member.id);
}

@Override
public int hashCode() {
    return Objects.hash(id);
}
```

## 3. 클래스와 인터페이스

점검 항목
* 접근 제어자를 최소한으로 두어 캡슐화했는가 (아이템 15)
* public 클래스에서 필드를 직접 노출하지 않고 접근자를 쓰는가 (아이템 16)
* 가변성을 최소화했는가, 가능하면 불변 객체로 두었는가 (아이템 17)
  불변 객체는 공유해도 안전하고 상태가 예측 가능하다.
* 상속보다 조합을 우선했는가 (아이템 18)
  상속은 상위 클래스 구현에 강하게 묶여 변경에 취약하다.
* 상속을 의도하지 않은 클래스는 확장을 금지했는가 (아이템 19)
  확장을 고려해 설계하고 문서화하거나, 아니면 final로 닫는다.
* 추상 클래스보다 인터페이스를 우선 고려했는가 (아이템 20)

```java
// 상속 대신 조합으로 결합을 낮춤 (아이템 18)
public class CountingSet<E> {
    private final Set<E> delegate;
    private int addCount;

    public boolean add(E e) {
        addCount++;
        return delegate.add(e);
    }
}
```

## 4. 제네릭

점검 항목
* 로 타입을 쓰지 않고 타입 파라미터를 명시했는가 (아이템 26)
  로 타입은 컴파일 시점 타입 검사를 잃는다.
* 비검사 경고를 제거했는가, 억제한다면 근거를 남겼는가 (아이템 27)
* 배열보다 리스트를 우선 고려했는가 (아이템 28)
* API 유연성을 위해 한정적 와일드카드를 적절히 썼는가 (아이템 31)
  생산자에는 extends, 소비자에는 super를 적용한다.

```java
// 로 타입 대신 타입 파라미터 명시 (아이템 26)
List<String> names = new ArrayList<>();
```

## 6. 람다와 스트림

점검 항목
* 익명 클래스 대신 람다로 간결하게 표현했는가 (아이템 42)
* 더 명확할 때 메서드 참조를 활용했는가 (아이템 43)
* 스트림을 적절한 곳에만 썼는가, 남용하지 않았는가 (아이템 45)
  모든 반복을 스트림으로 바꾸면 오히려 읽기 어렵다.
* 스트림 연산을 부수 효과 없이 작성했는가 (아이템 46)
  forEach로 외부 상태를 바꾸기보다 collect로 결과를 모은다.

```java
// 부수 효과 없이 결과를 수집 (아이템 46)
List<String> names = members.stream()
    .map(Member::getName)
    .collect(Collectors.toList());
```

## 7. 메서드

점검 항목
* 파라미터 유효성을 메서드 시작 부분에서 검사하는가 (아이템 49)
  잘못된 값은 들어오자마자 막아 원인에서 먼 곳에서 터지는 것을 방지한다.
* 필요한 경우 방어적 복사를 하는가 (아이템 50)
  외부에서 받은 가변 객체를 그대로 저장하면 호출자가 나중에 그 객체를 바꿔 내부 상태가 깨질 수 있다.
* null 대신 빈 컬렉션이나 빈 배열을 반환하는가 (아이템 54)
* 반환값이 없을 수 있으면 Optional을 적절히 활용하는가 (아이템 55)

```java
// null 대신 빈 컬렉션 반환 (아이템 54)
public List<Order> findByMember(Long memberId) {
    List<Order> result = orderRepository.findByMemberId(memberId);
    return result != null ? result : Collections.emptyList();
}
```
