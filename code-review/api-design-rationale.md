# API 설계 점검 항목의 근거 (Google AIP 기반)

이 문서는 [api-design-guideline.md](./api-design-guideline.md)의 핵심 점검 항목이 왜 필요한지를 구체적인 예시와 함께 설명한다.
원칙의 출처와 상세한 배경은 Google API Improvement Proposals(google.aip.dev)를 참고한다. 이 문서의 설명은 모두 새로 작성한 것이다.
예시의 경로와 필드 표기는 이해를 돕기 위한 REST 관점의 표현이다.

API는 한번 공개되면 사용자가 그 형태에 맞춰 코드를 작성하므로, 나중에 바꾸기가 매우 어렵다.
아래 근거는 대부분 "지금 잘 정하지 않으면 나중에 호환을 깨지 않고는 못 고친다"라는 점으로 모인다.

## 리소스 설계

### 왜 리소스(명사) 중심으로 설계하는가
동작(동사) 중심으로 API를 만들면 기능이 늘 때마다 새로운 엔드포인트가 무질서하게 증가한다.

```
# 점검 대상: 동사 중심 → 기능마다 엔드포인트가 제각각으로 늘어남
POST /createOrder
POST /cancelOrder
POST /getOrderList
POST /updateOrderAddress

# 개선: 리소스 중심 → 같은 표준 동작이 모든 리소스에 일관 적용됨
POST   /v1/orders
DELETE /v1/orders/{id}
GET    /v1/orders
PATCH  /v1/orders/{id}
```

사용자가 orders에서 배운 사용법을 members, products에 그대로 옮길 수 있다.

### 왜 API 표면을 DB 스키마와 분리하는가
API를 내부 DB 구조와 똑같이 노출하면, 표면이 내부 구현에 묶여 버린다.
예를 들어 응답에 내부 컬럼명 `mbr_no`, `ord_dt`를 그대로 내보내면, 나중에 테이블을 정규화하거나 컬럼명을 바꿀 때 사용자 코드가 깨져 바꿀 수 없게 된다.
응답을 별도 표현(`memberId`, `orderDate`)으로 두면 내부를 자유롭게 바꿔도 사용자에게 영향이 가지 않는다.

### 왜 순환 참조를 피하는가
리소스 사이에 순환 관계가 있으면 생성과 삭제 순서가 복잡해진다.
A와 B가 서로를 참조하면, A를 만들고 B를 만든 뒤 다시 A를 갱신해야 하는 식으로 절차가 늘어나고, 삭제 시에는 무엇을 먼저 끊어야 할지 따져야 한다.
비순환 구조로 두면 이 복잡성이 사라진다.

## 표준 메서드와 오퍼레이션

### 왜 표준 메서드를 우선하는가
표준 메서드는 사용자가 이미 알고 있는 동작이다.
같은 "조회"를 어떤 API는 `GET /orders/{id}`로, 어떤 API는 `POST /getOrder`로 표현하면 사용자가 매번 새로 배워야 한다.
표준에서 벗어난 커스텀 메서드는 표준으로 표현할 수 없을 때만(예: `POST /v1/orders/{id}:cancel`) 써야 표면이 예측 가능하게 유지된다.

### 왜 부분 수정에 필드 마스크를 쓰는가
필드 마스크 없이 전체 객체로 수정하면, 클라이언트가 일부만 바꾸려 해도 전체를 보내야 한다.

```
# 점검 대상: 이름만 바꾸려는데 전체 객체를 PUT
# → 클라이언트가 보내지 않은 phone 이 빈 값으로 덮어써질 수 있음
PUT /v1/members/123
{ "name": "kim" }     # phone 을 안 보냄 → 서버에서 phone 이 지워질 위험

# 개선: 바꿀 필드를 마스크로 명시 → 명시한 필드만 갱신
PATCH /v1/members/123?updateMask=name
{ "name": "kim" }
```

### 왜 오래 걸리는 작업을 장기 실행 오퍼레이션으로 처리하는가
요청 하나가 끝날 때까지 응답을 붙잡고 있으면, 대용량 내보내기 같은 작업에서 클라이언트와 서버 모두 자원을 오래 점유하고 타임아웃 위험이 커진다.
즉시 폴링 가능한 핸들(예: `{ "operationId": "abc", "done": false }`)을 반환하면, 클라이언트는 진행 상태를 확인하며 다른 일을 할 수 있다.

## 필드

### 왜 필드 이름에 전치사와 동사를 금지하는가
같은 개념을 가리키는 이름이 API마다 다르면 사용자가 한 API에서 익힌 것을 다른 API에 적용할 수 없다.

```
reason_for_error  →  error_reason     # 전치사 제거로 이름이 서로 맞을 확률이 높아짐
written_by        →  author
disable           →  disabled         # 필드는 동작이 아니라 값이므로 명사로
is_paid           →  paid             # boolean 의 is 접두어 제거
```

### 왜 민감 정보를 응답에 반환하지 않는가
비밀번호나 토큰 같은 값이 응답에 한 번이라도 실리면, 로그나 캐시, 중간 시스템을 거치며 통제 범위를 벗어나 노출된다.

```
# 점검 대상: 생성 응답에 passwordHash 까지 실림
POST /v1/users → { "id": 1, "email": "a@b.com", "passwordHash": "..." }

# 개선: 민감 필드는 입력으로만 받고 응답에서는 제외
POST /v1/users → { "id": 1, "email": "a@b.com" }
```

### 왜 필드 동작을 문서화하는가
어떤 필드가 필수인지, 출력 전용인지, 한번 정하면 못 바꾸는지를 사용자가 알 수 없으면, 서버가 채울 값(예: createTime)을 직접 보내거나 불변 필드를 바꾸려는 실수를 한다.
REQUIRED, OUTPUT_ONLY, IMMUTABLE 같은 동작을 명시하면 사용자가 무엇을 보내야 하고 무엇을 기대해야 하는지 분명해진다.

## 설계 패턴

### 왜 페이지네이션을 처음부터 넣는가
컬렉션은 시간이 지나며 커진다.

```
# 출시 시점: orders 가 30건이라 GET /v1/orders 가 전부 반환 → 사용자는 전체를 받는다고 가정
# 나중에 데이터가 늘어 기본 50건 페이지네이션을 추가하면:
#   전체를 받던 사용자 코드가 갑자기 50건만 받게 되고, 나머지를 못 가져온다.
```

이 변경은 기존 사용자의 동작을 깨고 되돌릴 수 없으므로, page_size, page_token, next_page_token을 처음부터 넣어야 한다.

### 왜 page_token을 불투명하게 두는가
페이지 토큰을 사용자가 해석할 수 있으면(예: `?pageToken=offset:50`), 사용자는 반드시 그 내부 구조에 의존하기 시작한다.
그 순간 토큰의 구현(오프셋 방식)이 API 계약의 일부가 되어, 커서 방식으로 바꾸면 사용자가 깨진다.
의미를 알 수 없는 불투명한 문자열로 두면 내부 구현을 자유롭게 바꿀 수 있다.

### 왜 etag로 신선도를 검증하는가
두 클라이언트가 같은 리소스를 동시에 읽고 각자 수정해 저장하면, 나중 저장이 앞선 저장을 덮어쓰는 갱신 손실이 생긴다.
요청에 "내가 본 버전(etag)"을 함께 보내고 서버가 현재 버전과 비교하면, 그사이 바뀐 경우 수정을 거부(412 Precondition Failed)해 손실을 막는다.

### 왜 요청 식별자로 멱등성을 보장하는가
네트워크 오류로 클라이언트가 같은 생성 요청을 재시도하면, 식별자가 없으면 같은 주문이 두 번 만들어진다.
요청에 고유한 request_id를 함께 보내면, 서버가 이미 처리한 id를 알아채고 중복 생성 대신 기존 결과를 돌려준다.

### 왜 소프트 삭제를 검토하는가
즉시 영구 삭제는 실수로 지운 데이터를 되돌릴 수 없게 만든다.
상태를 DELETED로 바꾸는 소프트 삭제는 복구(undelete) 여지를 남기고, 일정 기간 후 만료시키는 방식으로 안전과 정리를 함께 달성한다.

### 왜 인가를 존재 확인보다 먼저 하는가
권한이 없는 사용자에게 리소스의 존재 여부를 다른 응답으로 알려 주면, 사용자가 응답 차이로 리소스의 존재를 추측할 수 있다.

```
# 점검 대상: 존재를 먼저 확인 → 응답 차이로 정보가 새어 나감
#   /orders/999  없음 → 404
#   /orders/1000 있지만 남의 것 → 403
#   공격자는 403 이 나오는 id 가 "존재하는 주문"임을 알게 된다.

# 개선: 권한을 먼저 확인 → 권한 없으면 존재 여부와 무관하게 동일하게 403(또는 404)
```

## 호환성과 버전 관리

### 왜 호환을 깨는 변경을 새 메이저 버전으로 분리하는가
사용자는 현재 API 형태에 맞춰 코드를 작성해 두었다.
`total`을 `totalPrice`로 이름을 바꾸거나 타입을 string에서 int로 바꾸는 변경을 같은 버전에서 하면, 사용자의 파싱 코드가 예고 없이 깨진다.
새 메이저 버전(`/v2/...`)으로 분리하면 `/v1`을 쓰던 기존 사용자는 그대로 두고, 준비된 사용자만 옮겨 갈 수 있다.

### 왜 어떤 변경이 호환을 깨는지 미리 판단하는가
필수 필드 추가나 검증 강화처럼 겉보기에 사소한 변경도 기존 요청을 거부하게 만든다.

```
# 기존: { "name": "kim" } 으로 잘 동작하던 생성 요청
# 변경: phone 을 필수로 추가
# → 예전 형식으로 보내던 모든 클라이언트가 갑자기 400 을 받는다.
```

무엇이 호환을 깨는 변경인지 리뷰 단계에서 판단해야, 출시 후 사용자 코드가 깨지는 사고를 막는다.

## 오류

### 왜 표준 오류 구조와 기계 판독 식별자를 쓰는가
오류 형식이 제각각이면 클라이언트가 API마다 다른 오류 처리 코드를 작성해야 한다.

```json
// 점검 대상: 자유 형식 메시지 → 클라이언트가 문자열을 파싱해야 원인을 구분
{ "msg": "재고가 부족합니다 (상품 456)" }

// 개선: 표준 구조 + 기계 판독 식별자(reason, domain) → 문자열 파싱 없이 분기 가능
{
  "error": {
    "code": 409,
    "status": "FAILED_PRECONDITION",
    "message": "재고가 부족합니다.",
    "details": [{
      "@type": "type.googleapis.com/google.rpc.ErrorInfo",
      "reason": "OUT_OF_STOCK",
      "domain": "order.example.com",
      "metadata": { "productId": "456" }
    }]
  }
}
```

상태 코드(20여 종)만으로는 오류를 충분히 구분할 수 없으므로, reason과 domain으로 클라이언트가 메시지를 파싱하지 않고도 오류를 구분하게 한다.

### 왜 오류 메시지에 내부 정보를 담지 않는가
상세한 내부 오류 메시지는 사용자에게 도움이 안 될 뿐 아니라, 공격자에게 시스템 구조의 단서를 준다.

```json
// 점검 대상: SQL 과 테이블 구조가 그대로 노출
{ "message": "ERROR: duplicate key value violates unique constraint \"orders_pkey\"" }

// 개선: 해결 방향을 담은 간결한 메시지, 내부 정보는 로그에만
{ "message": "이미 처리된 주문입니다." }
```
